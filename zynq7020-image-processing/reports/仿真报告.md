# ZYNQ7020 图像处理课程设计仿真报告

## 1. sobel_00_rtl_sim 实验目标

`sobel_00_rtl_sim` 是本课程设计的第 0 个基础仿真实验，目标是在不依赖 Vivado 工程和开发板硬件的情况下，先验证 RGB 图像经 UART 帧输入、灰度转换、Sobel 边缘检测到视频流输出的 RTL 数据通路是否正确。

本实验验证的数据流如下：

```text
input_rgb.hex
  -> UART byte stream model
  -> uart_rx
  -> image_frame_rx
  -> rgb_to_gray
  -> sobel_core
  -> video_stream_model
  -> sobel_out.pgm / sobel_out.png
```

仿真重点包括：

- UART 字节接收和图像帧解析是否能正确识别帧头、行头、图像尺寸和 RGB888 像素数据。
- `rgb_to_gray` 是否能将 RGB888 图像转换为 8 bit 灰度数据。
- `sobel_core` 是否能完成 3x3 Sobel 边缘检测，并输出完整的 128x72 边缘图像。
- 系统在异常帧头、异常格式和错误行号输入时是否能产生 `frame_error`，在合法帧输入后是否能产生完整输出帧。

## 2. 输入图像格式和 UART 帧格式

### 输入图像格式

仿真输入图像由 `input_rgb.hex` 提供，图像尺寸为 128x72，格式为 RGB888。每个像素由 3 个字节组成，顺序为 R、G、B，每个颜色分量宽度为 8 bit。

| 项目 | 参数 |
| --- | --- |
| 图像宽度 | 128 像素 |
| 图像高度 | 72 像素 |
| 像素格式 | RGB888 |
| 每像素字节数 | 3 字节 |
| 有效像素数 | 9216 |
| 像素数据量 | 27648 字节 |

### UART 帧格式

UART 输入采用帧头加逐行数据的方式传输。仿真中一帧图像由帧头、每一行的行头和 RGB 像素数据组成。

帧头格式如下：

| 字段 | 字节内容 | 说明 |
| --- | --- | --- |
| 固定帧头 1 | `0x55` | 图像帧起始标志 |
| 固定帧头 2 | `0xaa` | 图像帧起始标志 |
| width_l | `0x80` | 宽度低字节，128 |
| width_h | `0x00` | 宽度高字节 |
| height_l | `0x48` | 高度低字节，72 |
| height_h | `0x00` | 高度高字节 |
| format | `0x18` | RGB888 格式标志 |

行头格式如下：

| 字段 | 字节内容 | 说明 |
| --- | --- | --- |
| 固定行头 1 | `0x33` | 行数据起始标志 |
| 固定行头 2 | `0xcc` | 行数据起始标志 |
| row_l | 行号低字节 | 当前行号 |
| row_h | 行号高字节 | 当前行号 |

每行行头之后跟随 128 个 RGB888 像素，即 128 x 3 = 384 字节。完整图像包含 72 行，因此一帧的有效 UART 数据量为：

```text
帧头 7 字节 + 行头 72 x 4 字节 + 像素 128 x 72 x 3 字节
= 7 + 288 + 27648
= 27943 字节
```

以上统计是协议层字节数，不包含 UART 8N1 传输中的起始位和停止位。

## 3. rgb_to_gray 和 sobel_core 功能说明

### rgb_to_gray

`rgb_to_gray` 模块负责将 RGB888 像素转换为 8 bit 灰度像素。模块输入包括 `rgb_valid`、`r`、`g`、`b` 以及当前像素坐标 `x`、`y`，输出包括 `gray_valid`、`gray` 和同步后的 `gray_x`、`gray_y`。

灰度计算采用定点近似公式：

```text
gray = (77 * R + 150 * G + 29 * B) >> 8
```

该公式近似于常见亮度转换公式：

```text
gray = 0.299R + 0.587G + 0.114B
```

模块对有效信号、灰度结果和像素坐标进行一级寄存，因此 `gray_valid` 相对于 `rgb_valid` 延迟 1 个时钟周期。

### sobel_core

`sobel_core` 模块负责对灰度图像执行 Sobel 边缘检测。模块内部使用两行缓存和 3x3 窗口寄存器形成卷积窗口，对每个像素计算水平方向和垂直方向梯度。

Sobel 卷积计算如下：

```text
Gx = -p00 + p02 - 2*p10 + 2*p12 - p20 + p22
Gy = -p00 - 2*p01 - p02 + p20 + 2*p21 + p22
```

其中 `p00` 到 `p22` 表示当前 3x3 邻域中的 9 个灰度像素。模块先求 `Gx`、`Gy` 的绝对值，再使用近似梯度幅值：

```text
mag = abs(Gx) + abs(Gy)
```

当 `mag` 大于 255 时，输出饱和为 255；否则输出低 8 bit 作为边缘强度。图像边界位置无法形成完整 3x3 窗口，因此边界像素输出为 0。最后一个像素输入后，模块会刷新尾部窗口并产生 `edge_frame_done`，经 `video_stream_model` 后表现为 `video_frame_done`。

### 遇到的问题和解决方法

1. 本机未检测到 Icarus Verilog / VVP。

   解决方法：使用脚本自动选择可用的 ModelSim 进行仿真。后续也使用 XSim 2023.2 做了交叉验证，输出图像 SHA-256 与 ModelSim 结果一致。

2. 完整 VCD 波形文件体积较大。

   解决方法：不直接提交原始 VCD 文件，而是保留关键波形截图 `exp00_key_waveform.png` 和 `exp00_key_waveform.svg`，报告中引用关键波形图片即可满足说明需要。

3. UART 按真实串口逐字节传输时仿真耗时较长。

   解决方法：仿真阶段使用 12 MHz 时钟和 1 Mbaud 波特率缩短仿真时间，同时保持 UART 帧格式和模块连接关系不变。该设置只影响仿真耗时，不改变 RGB888 帧解析和 Sobel 处理逻辑。

4. `edge_frame_done` 和 `video_frame_done` 容易混淆。

   解决方法：报告中统一把 `video_frame_done` 作为视频输出侧一帧完成标志说明。该信号由 `video_stream_model` 对 `sobel_core` 的 `edge_frame_done` 进行寄存同步后得到，含义是一帧 Sobel 输出已经完成。
