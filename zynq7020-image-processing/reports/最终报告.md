# ZYNQ7020 图像处理课程设计最终报告

## 1. 课程设计任务说明

本课程设计以黑金 ZYNQ7020 开发板为平台，完成一个由浅入深的 FPGA 图像处理系统。系统从 RTL 级 Sobel 仿真开始，逐步完成 HDMI 固定图显示、固定图 Sobel、PC 串口传图、PS 写入 AXI BRAM、PL 读取 BRAM 并 HDMI 显示、UART 输入图像 Sobel 显示，以及上位机命令控制显示模式。

根据老师原仓库要求，基础任务包括：

1. 完成 `sobel_00_rtl_sim` RTL 仿真，说明输入图像、输出边缘图和关键波形。
2. 完成 `sobel_01_hdmi_pattern` HDMI 固定图片显示，验证 HDMI 输出链路。
3. 完成 `sobel_02_hdmi_sobel` 固定图片 Sobel 显示，说明灰度转换和 Sobel 数据流。
4. 完成 `sobel_03_uart_hdmi` SDK 程序下载和运行，通过串口接收 PC 发送图像并写入 BRAM。
5. 使用 `host_camera_uart` 中的 GUI 或命令行脚本发送图像到 ZYNQ。
6. 完成 `sobel_04_uart_sobel_hdmi`，在 HDMI 上显示 UART 输入图像的 Sobel 边缘结果。
7. 完成 `sobel_05_pc_control_display`，通过 PC 控制 HDMI 显示模式、Sobel 阈值和边缘叠加。
8. 实验 0 到实验 4 每个实验至少完成 1 个基础扩展；实验 5 完成上位机控制显示能力。
9. 保存上板现象照片、串口输出、Vivado 资源利用率、时序和 DRC 结果。

第二周综合扩展选择老师给出的“综合扩展任务 1：基于 `sobel_05` 的上位机与输入规格扩展”。本设计采用方案 A：所有输入尺寸在 PC 上位机统一适配，最终映射为固定 `128×72 RGB888` 帧，保持 PS/PL/BRAM/HDMI 硬件链路不变。为便于验收和评分，大拓展已单独整理到 `大拓展_01_上位机输入规格扩展`，不与实验 5 基础工程混合。

在完成综合扩展任务 1（C 档，纯 PC 端输入规格扩展）的基础上，本组进一步完成了**综合扩展任务 3：增加 PL 图像处理算法——图像锐化（B 档）**，单独整理到 `大拓展_03_图像处理算法扩展`。任务 3 在实验 5 的 PL/PS 上新增锐化算法与实时强度控制，并提供与硬件逐位一致的软件参考。两个综合扩展互相正交、可在同一上位机工具中叠加演示（任务 1 负责输入缩放、任务 3 负责 PL 锐化），超出老师“三选一”的最低要求。任务 1 详见第 6 节，任务 3 详见第 6B 节。

## 2. 系统总体结构和数据流

系统整体分为 PC 上位机、ZYNQ PS、AXI BRAM、PL 图像处理和 HDMI 输出五个部分。

```mermaid
flowchart LR
    sim["实验0 RTL仿真\nUART模型+Sobel"] --> hdmi["实验1 HDMI固定图"]
    hdmi --> sobel["实验2 PL Sobel"]
    pc["PC上位机\n图片/摄像头/视频/控制命令"] --> uart["UART图像帧/控制帧"]
    uart --> ps["ZYNQ PS\n协议解析"]
    ps --> bram["AXI BRAM\n图像区+控制字"]
    bram --> pl["PL\n灰度/Sobel/显示选择"]
    pl --> out["HDMI显示\n原图/灰度/边缘/叠加"]
    sobel --> pl
```

基础数据流如下：

| 实验 | 输入来源 | 主要处理路径 | 输出现象 |
| --- | --- | --- | --- |
| 实验 0 | `input_rgb.hex` | UART 模型 -> 帧解析 -> `rgb_to_gray` -> `sobel_core` | PNG/PGM 边缘图和关键波形 |
| 实验 1 | ROM 固定图 | ROM -> HDMI 时序 -> `rgb2dvi` | HDMI 固定图放大显示 |
| 实验 2 | ROM 固定图 | ROM -> 灰度 -> Sobel -> 阈值二值化 | HDMI 黑白边缘图 |
| 实验 3 | PC UART 图像 | PC -> UART -> PS -> BRAM -> PL 读图 | HDMI 原图显示 |
| 实验 4 | PC UART 图像 | PC -> UART -> PS -> BRAM -> PL Sobel | HDMI 彩色边缘图 |
| 实验 5 | PC 图像 + 控制帧 | PC 控制 -> PS 控制字 -> PL 显示模式选择 | 原图/灰度/边缘/叠加切换 |
| 综合扩展 | 任意尺寸图片/目录/视频 | PC 端缩放、裁剪或补边 -> `128x72/160x90/144x108/64x36` 处理尺寸 -> 固定 128×72 帧 | 覆盖 C 档 3 种参考尺寸，不改硬件 |

UART 图像帧保持 RGB888 协议：

```text
frame header: 55 aa width_l width_h height_l height_h 18
line header : 33 cc row_l row_h
pixels      : R G B, repeated 128 times per row
```

实验 5 控制帧格式为：

```text
A5 5A cmd value
cmd=1: mode, cmd=2: threshold, cmd=3: overlay
```

关键实现思想如下：

```text
任意尺寸 BGR 输入
  -> prepare_frame(frame_bgr, width=128, height=72, fit_mode, content_size, fill)
  -> 固定 128×72 RGB888
  -> build_frame_packet
  -> UART 发送给原 sobel_05 链路
```

三种 fit 策略：

| 策略 | 含义 | 适用场景 |
| --- | --- | --- |
| `stretch` | 直接拉伸到 128×72，可能改变宽高比 | 保持旧工具行为，速度简单 |
| `letterbox` | 等比缩放，居中放置，空白处填充颜色 | 保持图像不变形 |
| `center-crop` | 等比放大后居中裁剪，铺满画面 | 无黑边显示，牺牲边缘区域 |

处理尺寸 `--proc-size` 支持 `128x72`、`160x90`、`144x108` 和 `64x36`。其中 `128x72`、`160x90`、`144x108` 对应 C 档评分表中的参考尺寸；`64x36` 保留为低处理尺寸对比。非 `128x72` 处理尺寸会作为上位机中间尺寸，最后统一映射回固定 `128×72` 发送帧，用于在不修改硬件的前提下展示输入规格和处理分辨率对画面效果的影响。

评分尺寸对应关系如下，本表是综合扩展 1 对 C 档要求的直接回应：

| 评分表 C 档参考尺寸 | 本仓库实现状态 | 实现方式 | 最终发送给 FPGA |
| --- | --- | --- | --- |
| `128x72` | 已支持 | 作为上位机处理尺寸，兼容旧版固定帧流程 | 固定 `128×72 RGB888` |
| `160x90` | 已支持 | 上位机先处理为 `160x90`，再映射回固定发送帧 | 固定 `128×72 RGB888` |
| `144x108` | 已支持 | 上位机先处理为 `144x108`，再映射回固定发送帧 | 固定 `128×72 RGB888` |

因此，扩展一不是只增加 `64x36` 对比效果，而是明确覆盖了评分表要求的三种 C 档参考尺寸；`64x36` 仅用于展示更低处理尺寸带来的块状显示差异。

## 6. 第二周综合扩展设计原理、仿真结果和上板结果

### 6.1 扩展题目和设计方案

选择综合扩展任务 1：基于 `sobel_05` 的上位机与输入规格扩展。采用方案 A：上位机统一完成尺寸适配，FPGA 接收侧保持固定 `128×72 RGB888`。

这样做的好处是：

1. 不修改 BRAM 地址映射，不改变 PS 接收协议。
2. 不修改 PL Sobel 处理尺寸，不影响 HDMI 放大逻辑。
3. 兼容实验 5 现有显示控制命令，包括原图、灰度、边缘、叠加和阈值控制。
4. 可以快速验证多输入尺寸、多输入来源和不同缩放策略。

### 6.2 离线验证结果

扩展离线测试覆盖 `640×480`、`1920×1080`、`1080×1920` 三种原始尺寸，三种 fit 模式和四种处理尺寸，形成 `3×3×4=36` 组验证。四种处理尺寸为 `128x72`、`160x90`、`144x108`、`64x36`，其中 `128x72`、`160x90`、`144x108` 三组就是评分表 C 档列出的参考尺寸，均已进入自动化矩阵验证。所有输出最终均为 `(72,128,3)` `uint8`，打包后帧长均为 `27943` 字节，帧头均为 `55 aa 80 00 48 00 18`。

- `center-crop` 能铺满输出画面，不引入黑边。
- C 档尺寸验收依据：`128x72`、`160x90`、`144x108` 三种参考尺寸全部实现并通过矩阵测试；`64×36` 仅作为额外对比尺寸。
- 控制帧仍为 `A5 5A cmd value`，不影响实验 5 的显示控制功能。

### 6.3 上板结果和最终演示截图

综合扩展的硬件侧复用实验 5 的 bitstream 与 PS 程序。由于扩展不修改 FPGA 处理尺寸，现场上板重点验证不同 fit 策略、`128x72/160x90/144x108/64x36` 处理尺寸与原 `sobel_05` 显示控制链路是否仍能稳定工作。

最终验收截图说明：第二周扩展能够在上位机端完成不同输入规格到固定 `128×72 RGB888` 的映射，并保持实验 5 的 FPGA 硬件链路不变。`stretch`、`letterbox`、`center-crop` 三种策略分别对应拉伸、补边和裁剪；`128x72`、`160x90`、`144x108` 处理尺寸覆盖 C 档参考要求，`64×36` 处理尺寸可以产生更粗的显示效果，但发送给 FPGA 的仍是固定 `128×72` 帧。

## 6B. 第二周综合扩展（二）：增加 PL 图像处理算法——锐化（任务 3 B 档）

### 6B.1 题目与设计方案

选择综合扩展任务 3：在 `sobel_05_pc_control_display` 的 PL 上新增一种图像处理算法，并通过上位机命令选择显示结果。本组实现**图像锐化（Laplacian / unsharp 增强）**，对应任务 3 的 **B 档**（新增 1 种算法 + 上板演示 + 与软件参考对比）。

设计取向是“最低风险地改 PL 加算法”：

1. **复用已验证窗口**：锐化所需的 4 邻域拉普拉斯与 Sobel 共用 `sobel_core.v` 里同一条已验证的 3×3 滑窗（中心像素 = `mid1`），只在其上**附加**一个输出端口 `lap_data`，Sobel 数据通路逐位不变，零回归风险。
2. **不动 BRAM / 时序 / BD**：不同于任务 1 的 B 档（改尺寸要扩 BRAM、搬控制寄存器），锐化只新增一块 `lap_mem` 与显示端一次乘加，不改 Block Design、BRAM 容量和 HDMI 时序。
3. **实时可调**：锐化强度 `k` 经控制帧 `A5 5A 04 k` 写入控制字 `0x900C`，PL 每帧读取并在显示端应用，因此**拖动上位机滑块即可让 HDMI 实时变锐，无需重发图像帧**——与实验 5 的 mode/threshold/overlay 控制完全同构。

### 6B.2 锐化定点算法（RTL / PS / 上位机三处逐位一致）

```text
gray  = (77*R + 150*G + 29*B) >> 8                  # 与 rgb_to_gray.v 一致
lap   = 4*center - up - down - left - right          # 4 邻域拉普拉斯，1 像素边界 = 0
delta = floor(strength * lap / 256)                  # RTL 算术右移 >>>8，与软件 >>8 / floor_divide 同义
out_c = clamp(c + delta, 0, 255)   for c in R,G,B    # 逐通道，保留彩色
```

`strength=0` 时输出与原图逐像素相同；强度越大边缘/纹理越锐，过大时高频处饱和裁剪（overshoot / halo，这是 Laplacian 锐化的固有特性，在真实照片上表现为自然的“变清晰”）。同一套整数运算在 PL（`hdmi_bram_sobel_display.v`）、PS（`main.c`）和上位机（`sharpen_algo.py`）三处实现，保证软件参考与硬件输出逐位一致。

### 6B.3 无板软硬件协同仿真验证（RTL == 软件 golden）

复用实验 5 的无板协同仿真链（真实上位机打包 → 真实 `main.c` PS 分发 → XSim 渲染 HDMI → 与软件 golden 逐像素比对），已扩展覆盖锐化，本地 Vivado 2023.2 XSim 全部通过：

| 验证项 | 结果 |
| --- | --- |
| 真实 `main.c` 收 `A5 5A 04 96` → 控制字 `0x900C` | `sharpen=96`，图像区 + 4 个控制字逐项一致 |
| RTL 自检（原 Sobel 四模式自洽） | `EXP05_SELFCHECK_TB=passed`，零回归 |
| 全分辨率 1280×720 逐像素对比 | **11 个配置全部 `=match`**，含 `sharp0/64/128/255`（`sharp0` 等于原图） |
| 上位机 numpy 预览 vs 软件 golden vs RTL | 三者逐位一致 |
| 协同仿真链 | `EXP05_COSIM_CHAIN=passed` |

这满足任务 3「完成 RTL 仿真，对比硬件输出和软件参考结果」的要求；软件参考即 `sharpen_algo.py`，与 GUI 实时预览同源。

### 7.2 性能分析

图像固定为 `128×72 RGB888` 时，每帧协议层字节数为：

```text
frame header 7B + line header 72×4B + pixel 128×72×3B
= 7 + 288 + 27648
= 27943B
```

若使用 UART 8N1 和 `115200` baud，理论传输时间约为：

```text
27943B × 10bit / 115200 ≈ 2.43s/frame
```

因此本系统的瓶颈主要在 UART 传输带宽，而不是 PL Sobel 本身。实验 5 和第二周扩展通过 PC 端预处理保持传输帧固定，可以避免硬件修改带来的时序风险，但不能突破 UART 带宽限制。若要提升实时性，后续可以考虑老师给出的网络传输扩展方向，使用 UDP/TCP 替代 UART。

### 8.2 总结

本课程设计完成了从 RTL 仿真到 HDMI 上板显示、从固定图像到 PC UART 输入图像、从单一 Sobel 输出到上位机控制显示模式的完整流程。基础实验验证了 RGB888 图像在 Verilog、UART、BRAM、PS/PL 和 HDMI 链路中的表示与传递方式，也验证了灰度转换、Sobel 边缘检测、阈值控制和彩色叠加的硬件实现。

第一周基础扩展完成了 HDMI 边框、Sobel 阈值对比、彩色边缘映射和显示模式控制。第二周综合扩展（一）选择上位机与输入规格扩展，在不修改硬件的前提下支持不同尺寸输入、三种 fit 策略和四种处理尺寸，其中 `128x72`、`160x90`、`144x108` 覆盖 C 档参考尺寸，并通过 12 项离线测试和 36 组矩阵验证保证 UART 帧格式和控制命令不变。最终上板演示能够展示原图、灰度、边缘和叠加模式，说明基础链路和控制链路可用。

在此基础上，本组又完成了第二周综合扩展（二）任务 3（B 档）：在 PL 上新增图像锐化算法，复用 Sobel 的 3×3 窗口附加 4 邻域拉普拉斯，不改 BRAM / 时序 / BD，强度经控制字 `0x900C` 实时可调；无板协同仿真已逐像素证明 RTL 与软件 golden 一致（11 配置含 4 档锐化全部 `=match`），OOC 综合时序达标（WNS +2.452 ns）。两个综合扩展正交、可在同一上位机工具叠加演示，锐化上板照片在现场验收时补充。

后续若继续改进，可以优先考虑两点：一是将 UART 传输升级为网络传输以提升帧率；二是从任务 3 的 B 档继续向 A 档演进——再增加 Prewitt/Laplacian 边缘、二值化等算法（`lap_mem` 与灰度已就绪，增量很小），凑齐“≥3 种算法”，并在加算法时同步优化流水线和资源，避免实验 5 已经较小的时序裕量继续收紧。
